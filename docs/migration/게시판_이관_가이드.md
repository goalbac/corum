# 레거시 홈페이지(hanwoolin.com) 게시판 이관 가이드

기존 홈페이지(구 ASP/MS SQL Server 기반, hanwoolin.com)의 게시판 데이터를 Corum으로 옮기기 위한
도구와 절차 정리. 이 문서는 사람뿐 아니라 다음에 이 작업을 이어받는 AI도 참고할 수 있도록
"왜 이렇게 만들었는지"와 "무엇을 이미 했는지"를 함께 남긴다.

## 1. 배경 / 요구사항

- 구 홈페이지는 게시판별로 테이블이 분리되어 있음 (예: `JwaNotice`/`JwaNoticeC`, `notice`/`noticec`).
  게시글 테이블과 댓글 테이블이 항상 쌍으로 존재하고, **컬럼 구조는 모든 게시판이 동일**하다.
- 구 사이트는 카테고리 개념이 없었고, Corum은 4depth 메뉴 중 일부를 카테고리로 재구성했다.
  단, 지금까지 이관한 게시판들은 옛 메뉴 리프가 Corum에서도 각각 독립된 게시판으로 존재해서
  카테고리 지정 없이 1:1 게시판 매핑만으로 충분했다.
- 최종적으로 dev에서 검증 후 같은 코드/쿼리를 운영 서버에도 그대로 실행할 수 있어야 한다.
- 원본 작성일/조회수/작성자명 등 이력을 그대로 보존해야 한다.

## 2. 아키텍처

코드 위치: `backend/src/main/java/com/corum/backend/migration/`

| 파일 | 역할 |
|---|---|
| `BoardMigrationRunner` | 실제 이관 로직. `@Profile("migration")`으로만 등록되는 `CommandLineRunner` |
| `BoardMigrationSpec` | 게시판 매핑 설정 1건 (구 게시글 테이블, 구 댓글 테이블, 대상 board_id) |
| `LegacyFtpClient` | 구 서버 FTP 접속/다운로드 헬퍼 |
| `LegacyDataSourceConfig`(`config` 패키지) | 구 DB(MS SQL Server)용 보조 데이터소스 |

설정 파일: `backend/src/main/resources/application-migration.yml` — **git에 커밋되지 않음**
(`.gitignore`에 등록됨, 실제 DB/FTP 접속정보가 들어있는 로컬 전용 파일이라 채팅에도 붙여넣지 않는 게 좋다).
아래 형태로 로컬에 직접 채워 넣고 사용한다.

```yaml
legacy:
  datasource:
    url: jdbc:sqlserver://<호스트>:1433;databaseName=<db명>;encrypt=true;trustServerCertificate=true
    username: <아이디>
    password: <비밀번호>
  ftp:
    host: <ftp호스트>
    port: 21
    username: <아이디>
    password: <비밀번호>
    root: /www

migration:
  dry-run: true   # true면 DB/S3에 실제로 쓰지 않고 무엇을 이관할지 로그만 출력
```

### 왜 raw SQL(JdbcTemplate)로 직접 INSERT하는가

Corum의 `BaseEntity`는 `@CreatedDate`로 `created_at`을 자동 관리하며 `updatable = false`다.
API나 JPA(`repository.save()`)를 거쳐 이관하면 원본 작성일이 무조건 "지금 시각"으로 덮어써진다.
그래서 게시글/댓글/파일 모두 `JdbcTemplate`으로 직접 INSERT하면서 `created_at`, `view_count` 등을
원본 값 그대로 넣는다.

### 재실행해도 안전한 이유 (멱등성)

`migration_id_map(legacy_table, legacy_oid, new_post_id)` 테이블에 "구 테이블명 + 구 OID -> 신규 ID"를
기록해둔다. 게시글 테이블명과 댓글 테이블명은 이름이 겹치지 않으므로(`JwaNotice` vs `JwaNoticeC`)
게시글/댓글 모두 같은 테이블을 키 네임스페이스로 재사용한다. 재실행 시 이미 기록된 항목은
건너뛰므로, dev에서 여러 번 테스트하거나 나중에 운영에 같은 코드를 실행해도 중복 삽입되지 않는다.

⚠️ **주의**: 이 멱등성 체크는 2026-07-01 작업 중 댓글에는 빠져 있었다가 뒤늦게 발견된 버그였다
(아래 "발견된 버그" 참고). 새로 비슷한 이관 로직을 추가할 때는 **게시글/댓글/파일 등 삽입되는
모든 엔티티마다 각각 멱등성 체크가 있는지** 반드시 확인할 것.

### 첨부파일 / 인라인 이미지 위치

- **첨부파일**: 게시글의 `FName`(저장명, 콤마로 여러 개 구분), `FName_`(원본명, 같은 인덱스로 대응),
  `FPath`(폴더명, 게시글마다 다를 수 있음 — 게시판 고정값이 아님)를 조합해서
  `{ftp root}/HBoard/{FPath}/{FName}` 경로에서 다운로드한다.
- **본문 인라인 이미지**: TipTap 이전 시절 위지윅 에디터가 `<img src="/UploadImages/2019-/...">`
  같은 절대경로를 본문 HTML에 그대로 박아 넣는다. 이 경로가 브라우저 URL 인코딩된 상태
  (`%ec%98%b9.jpg` 등)로 저장된 경우가 많아 **URL 디코딩 후** `{ftp root}` + 디코딩된 경로로 받는다.
  다운로드/업로드 성공 시 본문 속 URL을 `/api/files/inline/{새 파일명}`으로 치환한다.
- 파일은 다운로드 즉시 MinIO(S3 호환)로 올리고 로컬에는 남기지 않는다 (스트리밍 방식).

## 3. 발견된 버그와 수정 (전부 코드에 반영 완료)

1. **Postgres 기본 DataSource가 안 만들어짐** — `legacyDataSource`(MS SQL Server용) 빈을 추가하자
   Spring Boot의 `DataSourceAutoConfiguration`(`@ConditionalOnMissingBean(DataSource.class)`)이
   통째로 비활성화되어 Corum 기본 데이터소스 자체가 생성되지 않았다. `LegacyDataSourceConfig`에서
   `DataSourceProperties` + `DataSourceBuilder`로 Postgres용 DataSource/JdbcTemplate을 `@Primary`로
   직접 재정의해서 해결.
2. **Lombok이 `@Qualifier`를 생성자 파라미터에 안 옮김** — `@RequiredArgsConstructor` + 필드에 붙인
   `@Qualifier("legacyJdbcTemplate")`가 생성자로 복사되지 않아, `@Primary` 빈(Postgres)이 두 필드
   모두에 주입되는 문제가 있었다. `BoardMigrationRunner`의 생성자를 직접 작성해서 해결.
3. **FTP 한글 파일명 인코딩** — 구 서버가 Windows 계열이라 파일명을 UTF-8이 아닌 MS949(한글
   코드페이지)로 주고받는다. `LegacyFtpClient`에서 `setControlEncoding("MS949")`로 해결.
   추가로 본문 인라인 이미지 경로는 URL 퍼센트 인코딩까지 되어 있어서 디코딩 처리도 필요했다.
4. **댓글 중복 삽입 (2026-07-01 발견)** — 게시글은 `migration_id_map`으로 재실행 시 스킵됐지만,
   댓글은 이 체크가 빠져 있어서 이미 이관된 게시판을 포함한 전체 목록을 재실행할 때마다
   댓글이 매번 새로 삽입됐다. 2768건 게시판을 추가하면서 이전 3개 게시판(29/30/31)의 댓글이
   정확히 2배로 중복된 걸 발견 → 댓글도 `migration_id_map`에 기록하도록 수정, dev DB의 중복
   데이터는 삭제 후 재실행으로 정리함.

### ⚠️ 2026-07-09 발견: dev board_id ≠ prod board_id

dev와 prod는 메뉴 구성이 서로 달라(prod에만 있는 "큰스승님 홈페이지" 섹션 등) 같은 경로라도
board_id가 전부 다르다. `BoardMigrationRunner.SPECS`는 원래 dev board_id로 작성돼 있었는데,
**이 상태로 prod에 실행하면 엉뚱한 게시판에 데이터가 들어간다** (예: dev 13번 "알려드려요"는
prod에선 13번이 "한울글 알림방"). 2026-07-09에 `SPECS`를 위 표의 prod board_id로 전량 교체함
— 지금부터 이 러너는 **prod 데이터소스 대상으로만** 실행할 것. 앞으로 게시판을 새로 추가할
때도 dev가 아니라 반드시 prod DB에서 실제 board_id를 조회해서 넣을 것.

## 5. 실행 방법

```powershell
# JAVA_HOME 지정은 필수 (CLAUDE.md 참고)
$env:JAVA_HOME = "C:\Program Files\Java\jdk-21.0.1"

# 1) application-migration.yml에 실제 접속정보 채우기 (dry-run: true로 시작)

# 2) 8080은 보통 다른 백엔드가 떠있으니 다른 포트로 실행
.\gradlew bootRun --args='--spring.profiles.active=local,migration --server.port=8099'

# 3) 로그에서 [DRY-RUN] 항목/게시글·댓글 건수 확인 후 문제 없으면
#    application-migration.yml의 dry-run을 false로 바꾸고 다시 실행

# 4) 완료 후 서버 종료 (bootRun은 이관 후에도 계속 떠있으므로 Ctrl+C 또는 프로세스 종료 필요)
```

### 새 게시판 추가하는 법

`BoardMigrationRunner.SPECS` 리스트에 한 줄 추가하면 된다.

```java
new BoardMigrationSpec("구_게시글테이블명", "구_댓글테이블명", 대상_board_id)
```

- 대상 board_id는 Corum DB의 `boards` 테이블에서 확인 (또는 `menus` 테이블에서 메뉴 경로로 검색).
- 카테고리 지정이 필요한 게시판이면 `BoardMigrationSpec`에 category_id 필드를 추가하고
  `migratePosts`의 INSERT에 `category_id` 컬럼을 채우도록 확장하면 된다 (현재는 카테고리 없는
  게시판만 처리했음).

### 운영 서버에 실행할 때

같은 코드를 그대로 쓰되:
- `application-migration.yml`은 운영 서버 전용 값으로 별도 준비 (레거시 DB/FTP 접속정보는 dev와 동일할
  가능성이 높지만, Corum 쪽 접속 대상만 운영 `spring.datasource.*`/`storage.s3.*`로 바뀜)
- **운영 서버 배포는 사용자가 명시적으로 요청한 경우에만 진행** (CLAUDE.md 배포 규칙과 동일)
- dev에서 이미 검증된 게시판은 `migration_id_map`이 운영 DB엔 없으므로 처음부터 다시 돌아간다
  (dev와 운영은 별개 DB라 정상. 운영에서 한 번 더 dry-run으로 먼저 확인 권장)
